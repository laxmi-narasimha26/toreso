SYSTEM / ROOT RULES — TORESO ENTERPRISE BUILD STANDARD (READ BEFORE ANY CHANGE)

You are building “Toreso”: an enterprise-grade B2B packaging procurement marketplace (buyers/suppliers/admin/auditor/finance) with multi-plant, RFQ/quotes, orders, payments, logistics, audits/certifications, analytics. This is NOT a hobby project. Do NOT build a “vibe-coded” app. Every change must feel premium, secure, scalable, and production-ready.

ABSOLUTE NON-NEGOTIABLES
1) Enterprise quality only:
   - No placeholders, no TODOs, no “mock for now”, no half-finished flows, no loose ends.
   - No silent failures, no “catch and continue”, no inconsistent response formats.

2) Multi-tenant safety is a hard guarantee:
   - Never allow cross-organization data access. Default deny.
   - Every read/write is scoped by org_id (and plant_id/site when applicable) — never query tenant-owned entities by id alone.

   - Controllers/routes: thin request handling only.
   - Services: business rules, workflows, state transitions.
   - Repositories: DB access only (Prisma), always tenant-scoped.

4) Premium UX and maximum ease-of-access:
   - UI must look premium, purposeful, consistent, and easy to navigate (not random components).
   - Accessibility is required (keyboard nav, aria, contrast, focus states, error messaging).
   - Reduce cognitive load: clear information hierarchy, consistent spacing/typography, predictable navigation.
   - Excellent empty states, loading skeletons, inline validation, helpful microcopy.

BACKEND GOLD STANDARD (DO THIS EVERY TIME)
1) API Contract Consistency:
   - Versioned API base path must be consistent across envs (choose one canonical scheme and stick to it).
   - Response envelope ALWAYS: { success, message, data, meta? }.
   - Use correct HTTP codes (200/201/204/400/401/403/404/409/422/429/5xx).
   - List endpoints have stable pagination and sorting; validate/coerce query params.
2) Validation:
   - Validate body/query/params with Zod at the edge. Reject unknown fields unless explicitly allowed.
3) AuthN/AuthZ:
   - Every protected endpoint: authentication + authorization + org scoping.
   - RBAC is not just roles — enforce org membership and permissions; default deny.
   - Never trust client claims for org/role; verify from DB/cache.
4) Multi-tenant isolation:
   - All repository methods require org context (orgId, optionally plantId).
   - Prohibit “findUnique(id)” for tenant-owned objects unless compound key includes orgId.
   - Prefer RLS for defense-in-depth:
     - Set current tenant context per request (e.g., SET LOCAL) and enforce policies on tables.
     - If RLS not feasible for a specific table, explicitly justify and add additional guards + tests.
5) Transactions + workflows:
   - Use Prisma transactions for multi-step writes (order creation, order lines, inventory, payments).
   - Procurement workflows must use explicit state machines:
     - RFQ states, Quote states, Order states, Payment states, Shipment states.
     - Only allow valid transitions; log every transition.
6) Payments & webhooks (MUST be idempotent):
   - Do not store raw card data. Tokenized only.
   - Verify webhook signatures.
   - Implement idempotency keys for order finalization and webhook handling (dedupe table).
   - Prevent double-capture, double-fulfillment, duplicate order creation.
7) Reliable integrations:
   - Use Outbox pattern for “publish side effects” reliably:
     - DB transaction writes business changes + outbox row.
     - Worker processes outbox: send email/SMS, push webhooks, update search index.
   - External calls: strict timeouts, retries with backoff, circuit-breaker pattern where relevant.
8) Caching (no production footguns):
   - Never use Redis KEYS in prod paths. Use SCAN or versioned keys/tag invalidation.
   - Prevent cache stampedes (single-flight/locks/jittered TTL).
9) Observability:
   - Structured JSON logs with correlation/request IDs. Redact PII and secrets.
   - OpenTelemetry tracing (HTTP + DB + Redis + external calls).
   - Metrics for latency, error rate, queue depth, webhook failures, payment failures, indexing lag.
10) Security & compliance:
   - Helmet, strict CORS allowlist, rate limiting (especially auth), CSRF where relevant.
   - Secrets only via env/vault; validate config at boot and fail fast.
   - Tamper-evident audit logs for critical actions:
     - Auth events, org/user/role changes, product/pricing changes, RFQ/quote approvals, order state changes, payments/refunds/disputes, audit/certification updates.
   - Data retention/DSAR: design deletion/export flows (DPDP/GDPR).
